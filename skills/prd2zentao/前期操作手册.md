# prd2zentao 前期操作手册

> 用 Codex 把 PRD 同步到禅道前,先完成这些准备。内网禅道默认按「控制台粘贴脚本」模式执行。

---

## 一、每次同步前

### 1. 打开禅道并登录

- 在自己的 Chrome 浏览器里打开禅道,用禅道账号登录好。
- 保持目标产品页处于登录态。批量脚本依赖浏览器里的禅道会话 cookie,不需要把账号密码交给 Codex。
- 如果登录态过期,脚本可能返回 HTML 登录页;重新登录后再执行。

### 2. 确认目标产品

- 不同产品在禅道里是不同目录,搞错产品会导致需求散落。
- productID 从目标产品需求列表页 URL 获取:
  - 例:`http://你的禅道域名/index.php?m=product&f=browse&productID=75` → productID 是 `75`。
- 打通链路或试跑先用测试产品;正式同步前再次确认正式产品的 productID。

### 3. 确认目标模块和来源

- 模块只建到二级。PRD 的三级子菜单不映射成禅道三级模块。
- moduleID 从「模块维护」页面 URL 的 `currentModuleID=XX` 获取。
- 「来源」是禅道必填字段,具体取值要由产品经理确认;未确认时只能标注为暂用默认。

### 4. 准备 PRD

- PRD 第四章「产品范围」要稳定,需求只从这一章拆。
- 告诉 Codex:PRD 文件路径、目标产品(productID)、目标模块(moduleID 或需新建模块)、来源字段取值。

---

## 二、开跑时怎么说

一句话把关键信息给齐,例如:

> "把 `V1.x.x-配置核查/PRD-配置核查.md` 同步到禅道,产品是『禅道功能验证』(productID 75),模块挂到『配置核查』二级模块,来源暂用 customer。"

Codex 会先拆需求清单并让你确认粒度、优先级、来源。确认后再生成控制台粘贴脚本或在可用浏览器环境中执行。

---

## 三、控制台粘贴模式

内网禅道或自动访问被组织策略拦截时,默认走此模式:

1. Codex 生成一份 `*-控制台粘贴.js` 脚本,放在 PRD 同目录。
2. 你在已登录禅道的标签页里打开 `F12 → Console`,整段粘贴执行。
3. 执行完成后,把 `console.table` 的结果截图或文字贴回给 Codex。
4. Codex 按标题 / 动作 / 结果 / 信息逐条核对成功与失败原因。

不要让 Codex 用 curl、脚本或其他方式绕过内网访问策略。

---

## 四、需要你拍板的点

- **需求粒度**:一篇 PRD 拆成几条,由 Codex 给建议,你确认。
- **优先级**:默认值之外的优先级调整由你确认。
- **来源**:客户、项目团队、行业规范等归类由你或产品经理确认。
- **删除类操作**:删模块、删需求这类动作不由 Codex 代办。

---

## 五、常见问题

- **返回 HTML 登录页**:禅道掉登录了,重新登录后再跑。
- **评审人不能为空**:禅道开启了需求评审。默认带 `needNotReview=1`;若流程必须评审,先确认评审人。
- **建错产品了**:停下来,确认正确 productID,手工清理错建需求后再重跑。
- **重复需求**:脚本应按标题归一化后 upsert;发现重复时先暂停,核对标题差异。
